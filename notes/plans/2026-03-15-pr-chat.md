# PR Chat ŌĆö Interactive Conversation with the Reviewer

## Problem Statement

After Revu posts review comments on a PR, developers have no way to interact with the reviewer.
They can't ask "why did you flag this?", request clarification, or discuss findings. The review is
fire-and-forget. GitHub Copilot has the same limitation ŌĆö its code review is one-shot with no
conversational follow-up (a top community complaint). This is a differentiator opportunity.

## Decision Drivers

- ADO first, GitHub in a follow-up
- Context must be cheap ŌĆö chat replies should cost a fraction of a full review, not replay the
  entire review session
- Start simple ŌĆö give the chat agent a `ReviewSnapshot` (posted findings + summary + reviewed
  file manifest), not the review's process (tool call transcripts). The agent has tools to
  re-derive context on demand
- Same tool classes, different instance ŌĆö `ReviewerTools` with an empty `Diff` (no cache, no
  local search ŌĆö falls through to git connector). No Explore, no CodeGraph.
- Separate chat session ŌĆö keyed as `chat-{repoId}-{prId}`, distinct from the review session
  (`pr-{repoId}-{prId}`). Chat must not inherit the review's message history.
- Progressive enrichment ŌĆö start with `ReviewSnapshot` as context, later explore threading in
  condensed review traces if chat quality needs it

## Solution

New pipeline parallel to the review pipeline:

```
Reply on a Revu comment thread (ADO)
    ŌåÆ service hook (ms.vss-code.git-pullrequest-comment-event)
    ŌåÆ WebhookFunction.WebhookAdoComment (new method, route: webhook/ado/comment)
    ŌåÆ chat-queue
    ŌåÆ ChatFunction (new queue trigger)
        ŌåÆ Fetch thread to get properties + file context
        ŌåÆ Load ReviewSnapshot from store (graceful if absent)
        ŌåÆ Chat agent (ModelKey.Default, ReviewerTools with empty Diff)
        ŌåÆ Post reply on same thread via CreateCommentAsync
```

### Trigger: reply to a Revu thread (no @mention needed)

In ADO, any reply to a Revu-owned comment thread triggers the chat. Revu threads are already
tagged with `revu:version` in their `PropertiesCollection` ŌĆö this is the identity check. No
`@mention` needed: if you're replying to Revu's thread, you're talking to Revu.

For PR-level (non-thread) comments, require an explicit `@revu` mention.

### ADO comment webhook payload

Event type: `ms.vss-code.git-pullrequest-comment-event`

```json
{
    "eventType": "ms.vss-code.git-pullrequest-comment-event",
    "resource": {
        "comment": {
            "id": 2,
            "parentCommentId": 1,
            "author": { "displayName": "...", "id": "...", "uniqueName": "..." },
            "content": "Why did you flag this?",
            "publishedDate": "...",
            "lastContentUpdatedDate": "...",
            "commentType": "text",
            "isDeleted": false
        },
        "pullRequest": {
            "pullRequestId": 1,
            "sourceRefName": "refs/heads/feature",
            "targetRefName": "refs/heads/main",
            "status": "active"
        },
        "repository": {
            "id": "<guid>",
            "name": "Fabrikam",
            "project": { "name": "Fabrikam" }
        }
    }
}
```

**Key quirks:**
- No `action` field ŌĆö distinguish new vs edited by comparing `publishedDate` to
  `lastContentUpdatedDate`. Equal = new comment. Different = edit. Also check `isDeleted`.
- No thread ID in the payload ŌĆö only `comment.id` and `comment.parentCommentId`. To find the
  thread, we must list all PR threads and match by comment ID.
- No thread properties in the payload ŌĆö we need a separate API call to get the thread's
  `PropertiesCollection` (where `revu:version` and `revu:fingerprint` live).

### Self-reply loop prevention

When Revu posts a chat reply, ADO fires another comment webhook. Must filter these out.

Use a content marker: prefix every Revu chat reply with `<!-- revu:chat -->`. In
`ToChatRequest()`, check if `Comment.Content` starts with this marker ŌåÆ skip. ADO uses
`PropertiesCollection` for thread-level identity (`revu:version`, `revu:fingerprint`), but
properties aren't available in the webhook payload ŌĆö only `Comment.Content` is. A body marker
is the cheapest filter that works in the webhook without an API call. Author-based filtering
breaks when the PAT owner is also a developer (PAT posts as the developer, so author check
would filter out their legitimate questions too).

### Thread-to-finding linkage

When a user replies to a Revu finding thread:
1. Webhook gives us `comment.parentCommentId` (or just the comment in the thread)
2. We list PR threads, find the one containing this comment
3. Read its `revu:fingerprint` property
4. Match against findings in the stored `ReviewSnapshot`
5. Pass the matched finding + thread's `threadContext` (file path, line range) to the chat agent

This gives the chat agent precise context: "the user is asking about this specific finding on
this file at this line range."

**Summary thread replies:** The summary thread has `revu:version` but no `revu:fingerprint`.
`GetChatThreadContext` returns `Fingerprint = null`, `FilePath = null`. The chat agent gets the
full snapshot but no specific finding anchor ŌĆö it answers in the context of the overall review.

**PR-level `@revu` mentions:** No thread anchor at all. Same as summary thread ŌĆö full snapshot
context, no specific finding.

### Chat agent context

```
Chat agent receives:
Ōö£ŌöĆŌöĆ System prompt (conversational reviewer persona)
Ōö£ŌöĆŌöĆ ReviewSnapshot:                               ŌåÉ from ReviewStore
Ōöé   Ōö£ŌöĆŌöĆ ReviewResult (posted findings + summary)
Ōöé   Ōö£ŌöĆŌöĆ Reviewed files (path + change kind)       ŌåÉ knows what was in scope
Ōöé   ŌööŌöĆŌöĆ PR context (title, description)           ŌåÉ captured at review time, no re-fetch
Ōö£ŌöĆŌöĆ Thread anchor:                                ŌåÉ from GetChatThreadContext
Ōöé   Ōö£ŌöĆŌöĆ The specific finding (if reply to a finding thread, matched via fingerprint)
Ōöé   Ōö£ŌöĆŌöĆ File path + line range (from threadContext)
Ōöé   ŌööŌöĆŌöĆ Full ADO thread messages (one-shot, not merged into Cosmos history)
Ōö£ŌöĆŌöĆ Tools: FetchFile, SearchCode, ListDirectory, QueryCodeGraph (returns "not available")
ŌööŌöĆŌöĆ Chat history (prior agent turns via CosmosChatHistoryProvider, PR-scoped)
```

**Context deduplication:** ADO thread messages and Cosmos chat history overlap (both contain
prior exchanges). They serve different purposes: thread messages are the raw ADO conversation
(includes human comments the agent never saw), Cosmos history is the agent's own request/response
pairs (includes tool calls). To avoid the agent seeing its own replies twice, strip
`<!-- revu:chat -->` prefixed messages from the thread messages before injection ŌĆö the agent's
side of the conversation is already in Cosmos history.

### Chat agent

- Model: `ModelKey.Default` (fast, conversational ŌĆö not reasoning)
- Tools: `ReviewerTools` constructed with an **empty `Diff`** (`new Diff([])`) and no
  `CodeGraphQuery`. All four tool methods are exposed (can't selectively exclude from
  `ReviewerTools`). Behavior with empty Diff: `FetchFile` cache-misses fall through to git
  connector. `SearchCode` local-match returns nothing, remote search still works.
  `QueryCodeGraph` returns "No code graph available" (null guard at `ReviewerTools.cs:102`).
  This is intentional: chat is cheap, tools are escape hatches.
- No structured output ŌĆö free-form markdown response
- Max roundtrips: 3
- Chat history: keyed as `chat-{repoId}-{prId}`, persisted via `CosmosChatHistoryProvider`.
  PR-scoped, not thread-scoped ŌĆö the agent sees all chat turns across threads for this PR.

## Research Summary

**GitHub Copilot (March 2026):** Code review is fire-and-forget ŌĆö no conversational follow-up on
review comments. `@copilot` mention triggers the coding agent (makes code changes, opens child
PR), not a chat. Each review is stateless. Community discussions #166504, #174047, #168012 all
request conversational follow-up.

**ADO comment webhook:** See payload section above. Key additional note: service hook
subscription for `ms.vss-code.git-pullrequest-comment-event` must be configured separately
from the existing PR event hooks (ops step).

**Existing Revu infrastructure:**
- Revu threads already tagged with `revu:version` and `revu:fingerprint` properties ŌĆö
  `AdoConnector.cs:24-25, 218-226`
- `GitHttpClient.GetThreadsAsync` already used in `PostReview` for dedup ŌĆö
  `AdoConnector.cs:198-208`
- `GitHttpClient.CreateCommentAsync` exists in the SDK for posting to existing threads
- `CosmosChatHistoryProvider` persists sessions to Cosmos (180-day TTL) ŌĆö
  `ServiceCollectionExtensions.cs:40-50`
- `PrContextProvider` injects PR metadata + project config ŌĆö `Review/PrContextProvider.cs`
- `ReviewerTools` wraps IGitConnector methods as AI tools ŌĆö `Review/ReviewerTools.cs`
- `ReviewStore` persists audit events but not the `ReviewResult` itself ŌĆö
  `Infra/Cosmos/ReviewStore.cs`
- `WebhookFunction.Run` for ADO uses function key auth (no HMAC) ŌĆö
  `Functions/WebhookFunction.cs:28`
- `AdoWebhook` currently only handles `git.pullrequest.created/updated` ŌĆö
  `Git/AdoWebhook.cs:7`

## Implementation Steps

1. **Add `ReviewSnapshot` model and persist in ReviewStore**
   - Files: `Models.cs`, `Infra/Cosmos/ReviewStore.cs`, `Functions/ReviewFunction.cs`
   - New model in `Models.cs`:
     ```csharp
     record ReviewSnapshot(
         ReviewResult Result,
         IReadOnlyList<ReviewedFile> Files,  // what was reviewed
         string PrTitle,                     // captured at review time
         string? PrDescription               // avoids re-fetching on every chat turn
     );

     record ReviewedFile(string Path, ChangeKind Kind);
     ```
   - `ReviewSnapshot` is what the chat agent receives ŌĆö it knows the findings/summary, which
     files were in scope, and the PR context. This lets it answer "did you look at X?" or
     "why didn't you flag Y?" without re-deriving the diff or re-fetching PR metadata.
   - **Scope:** the snapshot captures the **post-cap** `ReviewResult` ŌĆö the inline-posted
     findings (top N by severity) plus the summary (which includes overflow findings as text).
     The pre-cap full list is internal to `Reviewer`. This is sufficient for v1: the summary
     text mentions overflow findings, and the agent has tools to investigate further.
   - `ReviewedFile` is deliberately minimal (path + change kind, no diffs or content) to keep
     the serialized snapshot small for prompt injection.
   - Add serialized `ReviewSnapshot` (JSON string) to `ReviewEvent`
   - Add `GetLatestSnapshotAsync(repositoryId, pullRequestId)` ŌåÆ returns `ReviewSnapshot?`
     from the most recent completed `ReviewEvent` (query: partition key `repositoryId`,
     `WHERE c.pullRequestId = @prId AND c.status = 'Completed' ORDER BY c.createdAt DESC`,
     top 1)
   - Add optional parameter to `SaveAsync`: `ReviewSnapshot? snapshot = null`. Existing call
     site in `ReviewFunction` (skipped/failed reviews) passes nothing ŌĆö no breaking change.
   - Update `ReviewFunction` to build snapshot from `ReviewResult` + `Diff.Files` + PR
     title/description (already fetched via `PrContextProvider`) and pass through

2. **Add ChatRequest model**
   - Files: `Models.cs`
   - ```csharp
     record ChatRequest(
         ReviewRequest Review,
         int CommentId,          // ADO comment ID
         string UserMessage      // comment content
     );
     ```
   - Composes `ReviewRequest` instead of duplicating its fields ŌĆö `ChatFunction` passes
     `req.Review` directly to `IGitConnector` tool methods (`GetFile`, `SearchCode`, etc.)
   - Deliberately minimal ŌĆö thread lookup, finding linkage, and file context happen in
     `ChatFunction`, not in the webhook

3. **Add ADO comment webhook model**
   - Files: `Git/AdoWebhook.cs`
   - New `AdoCommentWebhook` record matching the payload shape:
     ```csharp
     record AdoCommentWebhook(string EventType, AdoCommentResource Resource);
     record AdoCommentResource(AdoComment Comment, AdoPullRequest PullRequest, AdoRepository Repository);
     record AdoComment(int Id, int ParentCommentId, AdoAuthor Author, string? Content,
         DateTimeOffset PublishedDate, DateTimeOffset LastContentUpdatedDate,
         bool IsDeleted);
     record AdoAuthor(string Id, string DisplayName, string UniqueName);
     ```
   - `ToChatRequest()` method with filters:
     - `EventType == "ms.vss-code.git-pullrequest-comment-event"`
     - `!Comment.IsDeleted`
     - `Comment.Content` is not null/empty
     - `Comment.PublishedDate == Comment.LastContentUpdatedDate` (new only, not edits)
     - Content does not start with `<!-- revu:chat -->` (self-reply loop prevention)
   - Does NOT filter by `revu:version` ŌĆö that requires a thread lookup, too expensive in the
     webhook. `ChatFunction` does that check.

4. **Add `EnableChat` feature flag**
   - Files: `Infra/Options.cs`
   - Add `public bool EnableChat { get; init; }` to `RevuOptions` (default `false`)
   - Same pattern as `EnableIncrementalReviews` and `EnableCodeGraph`
   - Checked in the webhook function ŌĆö if disabled, drop the message before enqueuing

5. **Add ADO comment webhook function**
   - Files: `Functions/WebhookFunction.cs`
   - New function `WebhookAdoComment` on route `webhook/ado/comment` (separate from
     `webhook/ado`). Separate route because: different payload shape, different output queue,
     and ADO service hooks target a single URL per subscription ŌĆö can't multiplex.
   - Same auth (function key)
   - Inject `IOptions<RevuOptions>` ŌĆö check `EnableChat` early, return empty response if disabled
   - Deserialize as `AdoCommentWebhook`, call `ToChatRequest()`
   - New `ChatWebhookResponse` class with `[QueueOutput("chat-queue")]` binding
   - Repo store check (same as existing webhook ŌĆö only registered, enabled repos)

6. **Add thread lookup + reply to AdoConnector**
   - Files: `Git/AdoConnector.cs`, `Git/IGitConnector.cs`
   - New method on `IGitConnector`:
     ```csharp
     Task<ChatThreadContext?> GetChatThreadContext(ReviewRequest req, int commentId);
     Task PostChatReply(ReviewRequest req, int threadId, string body);
     ```
   - Takes `ReviewRequest` (not `ChatRequest`) to stay consistent with all other
     `IGitConnector` methods. `ChatFunction` passes `req.Review` and the chat-specific params.
   - `GetChatThreadContext`: list PR threads ŌåÆ find thread containing `commentId` ŌåÆ
     check `revu:version` property ŌåÆ if not Revu thread, check `@revu` mention
     (`content.Contains("@revu", StringComparison.OrdinalIgnoreCase)`) ŌåÆ
     return `ChatThreadContext` (threadId, fingerprint, filePath, line, thread comments) or
     null (ignore)
   - `PostChatReply`: `GitHttpClient.CreateCommentAsync` on the thread
   - `GitHubConnector`: stub both methods with `throw new NotSupportedException()` until
     GitHub chat is implemented
   - New model:
     ```csharp
     record ChatThreadContext(
         int ThreadId,
         string? Fingerprint,     // revu:fingerprint if it's a finding thread
         string? FilePath,        // from threadContext
         int? StartLine,
         IReadOnlyList<string> ThreadMessages  // ALL prior comments including Revu's own chat replies
     );
     ```
   - **Important:** `ThreadMessages` must include Revu's `<!-- revu:chat -->` replies ŌĆö the
     self-reply filter in `ToChatRequest()` prevents re-processing them as new requests, but
     the chat agent needs to see the full conversation chain for multi-turn context

7. **Build ChatFunction**
   - Files: new `Functions/ChatFunction.cs`
   - Constructor: `ChatFunction(IServiceProvider sp, IReviewStore reviewStore, ...)`
   - Queue trigger on `chat-queue`, deserialize `ChatRequest`
   - Resolve connector: `var git = sp.GetRequiredKeyedService<IGitConnector>(req.Review.Provider)`
     (same pattern as `ReviewFunction`)
   - **Parallel fetch** ŌĆö thread context and snapshot are independent:
     ```csharp
     var threadTask = git.GetChatThreadContext(req.Review, req.CommentId);
     var snapshotTask = reviewStore.GetLatestSnapshotAsync(req.Review.RepositoryId, req.Review.PullRequestId);
     await Task.WhenAll(threadTask, snapshotTask);
     ```
   - If thread context is null ŌåÆ silently drop (not a Revu thread and no `@revu` mention)
   - If snapshot is null (PR not yet reviewed) ŌåÆ proceed with degraded context (no
     findings/summary/file list, agent can still use tools to answer)
   - No config fetch ŌĆö chat doesn't filter files or apply review rules. The review's findings
     already reflect the config. PR context (title, description) is in the snapshot.
   - Build tools: `new ReviewerTools(git, req.Review, new Diff([]))` ŌĆö empty diff, no
     CodeGraph. Cache-miss on FetchFile falls through to git connector.
   - Build chat agent:
     - `ModelKey.Default` client
     - System prompt from `Prompts.ChatInstructions` (new)
     - Inject `ReviewSnapshot` (findings, summary, reviewed file list) + thread anchor
       (finding, file path, line range, thread messages) directly into the system prompt.
       No separate provider ŌĆö extract one later if this gets unwieldy.
     - Tools: FetchFile, SearchCode, ListDirectory from the `ReviewerTools` instance above
     - `CosmosChatHistoryProvider` with key `chat-{repoId}-{prId}`
   - Run agent with user message
   - Post reply via `git.PostChatReply(req.Review, threadContext.ThreadId, response)`

8. **Add chat prompt**
   - Files: `Review/Prompts.cs`
   - New `ChatInstructions` const ŌĆö conversational persona, aware it previously reviewed
     the PR, has findings available, should explain/clarify/discuss

9. **Register in DI**
   - Files: `Program.cs`
   - `WebhookFunction` needs `IOptions<RevuOptions>` ŌĆö add to constructor (already registered
     via `AddOptions<RevuOptions>`)
   - `ChatFunction` resolves `IGitConnector` via `IServiceProvider.GetRequiredKeyedService`
     at runtime (same pattern as `ReviewFunction`) ŌĆö no new registration needed
   - `ReviewerTools` is constructed inline in `ChatFunction` (not DI-resolved) ŌĆö no registration
   - `IChatClient`, `IReviewStore`, `CosmosChatHistoryProvider` are already registered
   - **Chat session keying:** the existing `CosmosChatHistoryProvider` singleton reads
     `SessionKeys.ConversationId` from StateBag (see `CoreStrategy.cs:77`). `ChatFunction`
     must set `session.StateBag.SetValue(SessionKeys.ConversationId, $"chat-{repoId}-{prId}")`
     before invoking the agent. This reuses the existing provider ŌĆö no new registration.

### Decided

- **Chat history scope:** PR-scoped (`chat-{repoId}-{prId}`). Thread messages from ADO are
  one-shot context injected into the system prompt, not merged into Cosmos chat history.
- **Thread properties on replies:** Post as `CommentType.Text`, add `revu:chat` to the
  thread's `PropertiesCollection` for future identification/metrics.

## Notes for Implementation

- ADO service hook for `ms.vss-code.git-pullrequest-comment-event` must be configured per
  project/repo alongside the existing PR event hooks. This is an ops step ŌĆö document it.
- The thread lookup in `GetChatThreadContext` calls `GetThreadsAsync` which fetches all threads.
  For PRs with many threads this could be slow. Future optimization: store `threadId` alongside
  each finding's fingerprint in the snapshot, then fetch a single thread by ID via
  `GetThreadAsync(threadId)` instead of listing all.
- `isDeleted` comments have null `Content` ŌĆö always null-check before accessing.
- **`@revu` mention matching:** case-insensitive substring match:
  `content.Contains("@revu", StringComparison.OrdinalIgnoreCase)`. No word-boundary check ŌĆö
  simple and sufficient. Applied in `GetChatThreadContext` when the thread has no `revu:version`
  property (i.e. not a Revu-owned thread). Top-level comments have `parentCommentId = 0` ŌĆö same
  thread lookup, same mention check. Returns `ChatThreadContext` with no fingerprint, no file anchor.
- **Composite index:** add a Cosmos composite index on `(pullRequestId ASC, createdAt DESC)` to
  the reviews container for efficient `GetLatestSnapshotAsync` queries.
- **Snapshot size:** always inject the full snapshot for v1. Post-cap findings are small (max
  ~5 inline + summary text + file manifest). No conditional trimming per context type.
