# Doc Watch — Automated Documentation PRs from Watched Repos

## Problem Statement

Teams maintain separate documentation repos (ADRs, architecture docs, API references) that drift
out of sync with code because updating them is a manual afterthought. In microservices setups,
N service repos all contribute to one central docs repo, making the drift problem worse.

The goal is an independent pipeline — separate from code review — that watches registered source
repos and opens PRs to a docs repo when something documentation-worthy changes.

## Decision Drivers

- Independent from code review — doc watch is its own pipeline with its own registration, not a
  second pass on the review pipeline. A repo can be watched for docs without being reviewed.
- N:1 topology — many source repos feed one docs repo. The agent reads the docs repo and figures
  out where content belongs. No manual section mappings.
- Registration via Cosmos — same pattern as `RepoStore` for review. A doc watch project is a
  Cosmos record: docs repo + list of source repos.
- The agent is smart enough to read the docs repo structure and decide where to place content,
  what to update, and whether a change is worth documenting.
- Chat works on both sides — source PR comments and doc PR comments.
- Signal over noise — only open PRs for genuinely documentation-worthy changes.
- Release Please batching model — one persistent PR per docs repo that accumulates changes, not
  one PR per source commit. GitHub is the state store.
- Docs repo has `.aporia.json` for style/tone/template preferences.
- Must work for both GitHub and ADO connectors.

## Research Summary

**Existing registration pattern** (`Aporia/Infra/Cosmos/RepoStore.cs`): `RepoStore` gates the
webhook — only registered repos trigger reviews. Doc watch needs a parallel store
(`DocWatchStore`) that maps source repos to their docs project. When a webhook arrives, Aporia
checks both stores independently.

**Pipeline structure** (`Aporia/Functions/ReviewFunction.cs`): Review pipeline is
`GetConfig → GetDiff → Review → PostReview`. Doc watch is a separate pipeline, not coupled to
this. It shares the webhook entry point but routes to its own queue and function.

**IGitConnector gap** (`Aporia/Git/IGitConnector.cs`): Currently read-only + comment posting.
Doc watch needs write operations: create branch, upsert files, create/update PR — on a different
repo than the webhook source.

**Strategy/tool pattern** (`Aporia/Review/CoreStrategy.cs`, `ReviewerTools.cs`): CoreStrategy
uses a reasoning model with tools (FetchFile, SearchCode, ListDirectory, Explore). The doc watch
agent reuses this pattern — tools pointed at both source and docs repos.

**Chat support** (`Aporia/Functions/ChatFunction.cs`): Existing chat pipeline handles PR comment
webhooks. Doc watch PRs need the same treatment — comments on doc PRs route to the doc watch
agent with context about both the source change and the docs repo.

**Release Please pattern** (from research): Google's Release Please uses GitHub as the state
store. No queue, no database for batching. On each trigger it re-derives all unreleased changes
since the last tag/marker, then creates or updates a single persistent PR. The PR branch and
body encode state. Batching is implicit — multiple commits between human merges accumulate in
the same PR. This is the model for doc watch: one open PR per docs repo that the agent keeps
updating as source changes land.

**Prior art** (from research):
- *Mintlify Autopilot*: cross-repo, AI-judged, creates draft PRs. Validates the approach of
  letting the agent read target repo structure to decide what to update.
- *Swimm Auto-sync*: three-tier staleness classification (up-to-date / out-of-sync / outdated).
  Good noise-reduction model.
- *Renovate*: best noise-reduction playbook — scheduling, PR grouping, rate-limiting, automerge.
- *Claude Code PR Autodoc Action*: bot-exclusion filter to prevent infinite loops. Essential.
- **Gap**: No existing tool auto-detects architectural decisions from code changes and opens PRs
  to a separate docs repo. This is novel.

## Solution

### Registration

A doc watch project is a Cosmos record registered via API:

```csharp
public record DocWatchProject(
    string DocsRepo,              // "org/platform-docs" — where PRs get opened
    List<string> SourceRepos,     // ["org/payments-service", "org/auth-service", ...]
    GitProvider Provider,
    string Organization,
    long? InstallationId          // GitHub App installation (for auth on docs repo)
);
```

### Docs Repo Config

The docs repo has an `.aporia.json` with preferences the agent follows:

```json
{
  "docWatch": {
    "context": "This is the platform architecture docs for our microservices system.",
    "diagrams": { "enabled": true, "format": "excalidraw" }
  }
}
```

No section mappings, no trigger rules. Just context and preferences. The agent reads the repo
structure and existing docs to understand where things go and what style to follow.

### Webhook Routing

```
Webhook from org/payments-service (PR merged)
        ↓
WebhookFunction:
  RepoStore lookup → registered for review? → enqueue to review-queue
  DocWatchStore lookup → is this a source for any doc watch project? → yes
        ↓
  Enqueue DocWatchRequest to docwatch-queue (one per matching project)
```

Both lookups happen independently. A repo can be in both, either, or neither.

### Doc Watch Pipeline (Release Please Model)

```
DocWatchFunction (queue trigger on docwatch-queue)
        ↓
  Fetch diff from source repo
  Fetch .aporia.json from docs repo (context, diagram prefs)
        ↓
  Agent reads docs repo structure (ListDirectory, FetchFile)
  Agent reads the diff + PR context from source repo
        ↓
  Agent decides: is this worth documenting?
    NO  → exit
    YES → agent generates content, decides where it goes
        ↓
  Search for existing open doc watch PR (by label "docwatch" on docs repo)
    FOUND  → push new commits to existing PR branch, update PR body
    NOT FOUND → create branch, upsert files, open new PR
        ↓
  Post comment on source PR linking to doc PR
```

**Batching is implicit.** If 5 source PRs merge before anyone reviews the doc PR, the agent
updates the same PR 5 times — each run adds its changes as new commits on the PR branch. The
PR body accumulates a changelog of which source PRs contributed. When a human merges the doc PR,
all accumulated changes land together.

The PR body encodes state — which source PRs have been processed, what was generated. Each run
reads the existing PR to understand what's already there and avoids duplicating work.

### Chat on Doc PRs

Comments on doc watch PRs route through the existing chat pipeline. The chat agent gets context
about the doc watch project (which source PRs triggered it, what changed) so it can answer
questions about both the documentation and the underlying code changes.

### Diagram Generation

When the change is architectural (new component, changed data flow, new integration), the agent
produces Excalidraw JSON files alongside markdown — simple C4-style diagrams (context, container,
component levels). Excalidraw format is version-controllable JSON, diffable, and renderable in
GitHub/IDEs with plugins.

### Loop Prevention

Doc watch skips its own PRs: branch names prefixed with `docwatch/` and PR titles prefixed with
`[Doc Watch]` are filtered in webhook processing. Bot user type as secondary guard.

## Implementation Steps

1. **Add DocWatchStore (Cosmos)**
   - Files: `Aporia/Infra/Cosmos/DocWatchStore.cs` (new)
   - Same pattern as `RepoStore` — thin Cosmos-backed persistence
   - `DocWatchProject` record: `DocsRepo`, `SourceRepos`, `Provider`, `Organization`,
     `InstallationId`
   - Lookup method: `GetProjectsBySource(provider, sourceRepo)` — returns all doc watch projects
     that list this repo as a source. This is the hot path called on every webhook.
   - Partition key on `Provider + Organization` (same pattern as RepoStore)

2. **Add registration API endpoint**
   - Files: `Aporia/Functions/DocWatchFunction.cs` (new)
   - HTTP-triggered endpoint to register/update/delete doc watch projects
   - Validates inputs, writes to `DocWatchStore`

3. **Add write operations to IGitConnector**
   - Files: `Aporia/Git/IGitConnector.cs`, `Aporia/Git/GitHubConnector.cs`,
     `Aporia/Git/AdoConnector.cs`
   - New methods: `CreateBranch(repo, baseBranch, newBranch)`,
     `UpsertFile(repo, branch, path, content, commitMessage)`,
     `CreatePullRequest(repo, sourceBranch, targetBranch, title, body)`,
     `UpdatePullRequest(repo, prId, title, body)`,
     `FindOpenPullRequest(repo, label)` — searches for existing doc watch PR
   - These operate on an arbitrary repo (not just the webhook source), so they take a repo
     identifier parameter
   - GitHub: Octokit `GitHubClient.Git.Reference`, `Repository.Content`,
     `PullRequest.Create`, `PullRequest.Update`, `Issue.Labels`
   - ADO: equivalent REST API calls

4. **Add DocWatchRequest model and docs repo config**
   - Files: `Aporia/Models.cs`, `Aporia/ProjectConfig.cs`
   - `DocWatchRequest` record: `GitProvider`, `SourceRepo`, `PullRequestId`, `SourceBranch`,
     `TargetBranch`, `DocsRepo`, `Organization`, `InstallationId`
   - Add `DocWatchConfig` to `ProjectConfig` with `Context` (string) and `Diagrams` settings
   - Parsed from docs repo `.aporia.json` at runtime

5. **Route webhooks to docwatch-queue**
   - Files: `Aporia/Functions/WebhookFunction.cs`
   - After existing review-queue enqueue logic, independently call
     `DocWatchStore.GetProjectsBySource()` for the webhook source repo
   - For each matching project, enqueue a `DocWatchRequest` to `docwatch-queue`
   - Add loop prevention: skip if branch starts with `docwatch/` or title starts with
     `[Doc Watch]`

6. **Build DocWatchFunction (queue trigger)**
   - Files: `Aporia/Functions/DocWatchFunction.cs`
   - Queue trigger on `docwatch-queue`
   - Fetches diff + PR context from source repo via `IGitConnector`
   - Fetches `.aporia.json` from docs repo for preferences
   - Runs doc watch agent
   - If agent returns content:
     - Search for existing open PR with `docwatch` label on docs repo
     - If found: push new commits to existing branch, update PR body with new changelog entry
     - If not found: create branch (`docwatch/pending`), upsert files, open new PR with
       `docwatch` label
   - Post comment on source PR linking to the doc PR (two-way traceability)

7. **Build doc watch agent**
   - Files: `Aporia/DocWatch/DocWatchAgent.cs` (new)
   - Same pattern as `CoreStrategy`: reasoning model + structured output + tools
   - Tools pointed at both repos:
     - Source repo: `FetchFile`, `ListDirectory`, `SearchCode` (understand the change)
     - Docs repo: `FetchFile`, `ListDirectory`, `SearchCode` (understand existing docs)
   - Agent receives: diff, PR context, docs repo config (context + diagram prefs), existing
     doc watch PR body (if updating — so it knows what's already been processed)
   - Agent decides: is this worth documenting? If yes, what to create/update and where?
   - Structured output: `DocWatchResult { ShouldUpdate, FilesToCreate, FilesToUpdate, Diagrams,
     ChangelogEntry }` where each file has path + content, diagrams are Excalidraw JSON (C4
     style), and changelog entry summarizes what this source PR contributes
   - No hardcoded rules — the agent reads both repos and uses judgment

8. **Wire up chat for doc watch PRs**
   - Files: `Aporia/Functions/ChatFunction.cs`
   - When a comment comes in on a PR with a `docwatch/` branch, route to doc watch chat handler
   - Chat agent gets context: which source PRs triggered this, the diffs, the docs repo structure
   - Uses same tools as the doc watch agent (both repos)

9. **Register in DI**
   - Files: `Aporia/Program.cs`
   - Register `DocWatchStore`, `DocWatchAgent` as scoped services
   - Register queue output binding for `docwatch-queue`

## Open Questions

- [ ] PR branch strategy — one persistent branch `docwatch/pending` that accumulates, or
      per-source branches that get superseded? Leaning toward single persistent branch (Release
      Please model).
- [ ] Conflict resolution — if the human edits the doc watch PR before merging (fixing tone,
      adding context), and then a new source PR triggers an update, the agent needs to not
      overwrite the human edits. Agent should read the current PR branch state, not just the
      main branch.
- [ ] Excalidraw C4 diagram quality — start with simple box-and-arrow component/container
      diagrams. Iterate on layout quality. May need a post-processing step for positioning.
- [ ] What happens when the doc watch PR is merged? Should the agent tag/label it so the next
      run knows to start a fresh PR? Or just search for open PRs (merged = gone = create new)?
