# Centralized Session & OTEL Store

## Problem Statement

Session data and OTEL traces are disconnected with no join key. In tests, sessions are JSON files
on disk (`FileSessionProvider`); in prod, MAF's `CosmosChatHistoryProvider` stores individual
messages but not the full envelope (agent name, instructions, tools). OTEL goes to OpenObserve
locally and Azure Monitor in prod. There is no way to look up a past review and see its sessions +
OTEL together.

## Decision Drivers

- No duplication — don't store messages twice
- Reuse MAF's `CosmosChatHistoryProvider` and its `"sessions"` container
- Full envelope metadata (agent, instructions, tools, traceId, strategy, version) needed for
  analysis alongside the messages MAF already stores
- `FileSessionProvider` stays for local test runs (no Cosmos dependency)
- OTEL dual export (OTLP + Azure Monitor) already works via env vars — no code change needed

## Solution

Subclass `CosmosChatHistoryProvider` to create `RevuChatHistoryProvider`. Override
`StoreChatHistoryAsync`: call `base` (MAF stores messages as individual documents), then write one
lightweight **metadata document** to the same `"sessions"` container. The metadata document
contains agent name, instructions, tools, traceId, strategy, version, and a timestamp — but NOT
messages (those are already stored by MAF). Use `type: "SessionEnvelope"` to distinguish from
MAF's `type: "ChatMessage"` documents. Same partition key (`/conversationId`), same TTL.

Store `traceId`, `strategy`, and `version` on `ReviewEvent` as the join key between reviews,
sessions, and OTEL.

```
CosmosChatHistoryProvider (MAF — stores individual messages)
  └── RevuChatHistoryProvider : CosmosChatHistoryProvider
        override StoreChatHistoryAsync:
          1. await base.StoreChatHistoryAsync(...)   // MAF writes messages
          2. Write metadata-only envelope document   // we add context
```

```
"sessions" container (partitioned by /conversationId):
  ┌─────────────────────────────────────────────────────────┐
  │ MAF documents (type: "ChatMessage")                     │
  │   { conversationId, role, message, timestamp, ttl }     │
  │   one per message, written by base provider             │
  ├─────────────────────────────────────────────────────────┤
  │ Our documents (type: "SessionEnvelope")                 │
  │   { conversationId, traceId, sequence, agent,           │
  │     instructions, tools, strategy, version,             │
  │     createdAt, ttl }                                    │
  │   one per agent invocation, metadata only               │
  └─────────────────────────────────────────────────────────┘
```

## Research Summary

### MAF ChatHistoryProvider extensibility
**Source:** `/Users/ivan/dev/agent-framework/dotnet/src/Microsoft.Agents.AI.Abstractions/ChatHistoryProvider.cs`

Key virtual methods:
- `StoreChatHistoryAsync(InvokedContext, CancellationToken)` — called from `InvokedCoreAsync`
  after agent invocation completes. Override point for persistence.
- `InvokedCoreAsync(InvokedContext, CancellationToken)` — full post-invocation lifecycle.
- `InvokingCoreAsync(InvokingContext, CancellationToken)` — pre-invocation.

Class is **not sealed** — subclassing is the intended extension pattern (see
`BoundedChatHistoryProvider` sample in MAF repo).

### CosmosChatHistoryProvider internals
**Source:** `/Users/ivan/dev/agent-framework/dotnet/src/Microsoft.Agents.AI.CosmosNoSql/CosmosChatHistoryProvider.cs`

- Stores individual `CosmosMessageDocument` per message with `type: "ChatMessage"`
- Partitioned by `/conversationId`
- TTL via `MessageTtlSeconds`
- State initialized via `Func<AgentSession?, State>` constructor delegate
- Not sealed — subclassing viable

### Getting envelope data from InvokedContext
**Source:** `tests/Revu.Tests.Integration/Fixtures/FileSessionProvider.cs`

`FileSessionProvider` already extracts everything we need:
- **Agent name:** `context.Agent is ChatClientAgent ca ? ca.Name : null`
- **Instructions:** `context.Agent is ChatClientAgent ca ? ca.Instructions : null`
- **Tools:** `context.Agent.GetService<ChatOptions>()?.Tools?.OfType<AIFunctionDeclaration>()`

Cast to `ChatClientAgent` for name/instructions, service discovery for `ChatOptions` → tools.

### Current trace/session data flow
- `ConversationId` = `$"pr-{RepositoryId}-{PullRequestId}"` (from `Models.cs`)
- `TraceId` auto-generated by OpenTelemetry — not stored anywhere persistent
- `ReviewStore` persists `ReviewEvent` with `conversationId` but no `traceId`

### OTEL export config
**File:** `Revu/Infra/Telemetry/TelemetryExtensions.cs`

Already supports dual export — `UseOtlpExporter()` and `UseAzureMonitorExporter()` can run
simultaneously. Controlled by env vars `OTEL_EXPORTER_OTLP_ENDPOINT` and
`APPLICATIONINSIGHTS_CONNECTION_STRING`.

## Implementation Steps

1. **Create `RevuChatHistoryProvider`**
   - Files: `Revu/Infra/AI/RevuChatHistoryProvider.cs`
   - Subclass `CosmosChatHistoryProvider`
   - Override `StoreChatHistoryAsync`:
     - Call `base.StoreChatHistoryAsync(...)` (MAF writes messages)
     - Build metadata envelope: agent name, instructions, tool definitions (names + schemas),
       traceId from `Activity.Current?.TraceId`, strategy, version
     - Write as a single document to the same `"sessions"` container with
       `type: "SessionEnvelope"`, same `conversationId` partition
   - Get agent data using `FileSessionProvider`'s proven pattern: cast to `ChatClientAgent`,
     `GetService<ChatOptions>()`
   - Sequence: `Interlocked.Increment` on a `ConcurrentDictionary<string, int>` keyed by
     conversationId
   - Strategy: passed via session `StateBag` (set in `CoreStrategy` before agent runs)
   - Version: `Assembly.GetExecutingAssembly().GetCustomAttribute<AssemblyInformationalVersionAttribute>()?.InformationalVersion`

2. **Wire `RevuChatHistoryProvider` in DI**
   - Files: `Revu/Infra/ServiceCollectionExtensions.cs`
   - Replace `CosmosChatHistoryProvider` registration with `RevuChatHistoryProvider`
   - Needs the `CosmosClient` + container name (already available from `CosmosOptions`)

3. **Add `traceId`, `strategy`, `version` to `ReviewEvent`**
   - Files: `Revu/Infra/Cosmos/ReviewStore.cs`, `Revu/Functions/ReviewFunction.cs`
   - Add `TraceId`, `Strategy`, `Version` properties to `ReviewEvent`
   - Capture `Activity.Current?.TraceId.ToString()` in `ReviewFunction.Run()`
   - Strategy from the strategy key used by `Reviewer`
   - This is the index — look up any review, get its traceId, query sessions + OTEL

4. **Prod OTEL → OpenObserve (alongside App Insights)**
   - Files: deployment/infra config (not application code)
   - Set both `OTEL_EXPORTER_OTLP_ENDPOINT` (OpenObserve) and
     `APPLICATIONINSIGHTS_CONNECTION_STRING` (App Insights) in prod
   - Already supported, zero code changes

## Notes for Implementation

### OpenObserve setup — prod
- Deploy OpenObserve instance accessible from Azure Functions app
- OTLP HTTP ingestion on port 5081, UI/API on port 5080
- Set `OTEL_EXPORTER_OTLP_ENDPOINT=https://<openobserve-host>:5081` in app settings
- Auth via `OTEL_EXPORTER_OTLP_HEADERS=Authorization=Basic <base64>`
- Keep `APPLICATIONINSIGHTS_CONNECTION_STRING` for App Insights as secondary
- Retention: match session TTL (180 days) or longer

### OpenObserve setup — local
- Already running locally (`verify.py` connects to `http://localhost:5080`)
- Set `OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:5081` in `local.settings.json`
- Default creds: `root@example.com` / `Complexpass#123` (matches `verify.py`'s Basic auth)
- Docker: `docker run -d -p 5080:5080 -p 5081:5081 -e ZO_ROOT_USER_EMAIL=root@example.com -e ZO_ROOT_USER_PASSWORD=Complexpass#123 public.ecr.aws/zinclabs/openobserve:latest`
- Parameterize `verify.py`'s OpenObserve URL and auth via env vars (`OPENOBSERVE_URL`,
  `OPENOBSERVE_AUTH`)

### Querying sessions later
- Envelope metadata: query `"sessions"` container with `WHERE type = 'SessionEnvelope' AND traceId = '...'`
- Messages: query same container with `WHERE type = 'ChatMessage' AND conversationId = '...'`
  (or use MAF's `ProvideChatHistoryAsync` directly)
- OTEL: query OpenObserve by traceId (same API `verify.py` already uses)
- `verify.py` remote mode is a future follow-up — for now, Cosmos Data Explorer and OpenObserve
  UI cover manual inspection
